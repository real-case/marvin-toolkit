Analyze staged git changes and generate a conventional commit message. Follow the full workflow below.

## Workflow

### 1. Check State

Run in parallel:

```bash
git status
git diff --stat
git diff --cached --stat
git log --oneline -10
git diff
git diff --cached
```

If no changes exist (nothing staged, unstaged, or untracked) — inform the user and stop.

### 2. Stage Changes

Prefer staging specific files by name over `git add -A` or `git add .`.

After staging, detect sensitive files:

```bash
git diff --cached --name-only | grep -iE '\.(env|pem|key|p12|pfx)$|credentials|secret|token|id_rsa|id_ed25519|npmrc|pypirc'
```

If matches found:
1. Unstage them: `git reset HEAD <file>`
2. Warn the user
3. Do not proceed until resolved

Confirm staged content with `git diff --cached --stat` before continuing.

### 3. Generate Message

If a message was provided via `$ARGUMENTS`, use it directly — skip to step 4.

Otherwise, analyze the staged diff and generate a message per the format below.

**Commit message format:**

```
<type>(<scope>): <subject>

<body>

<footer>
```

- **Type:** `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `ci`, `perf`, `style`, `build`
- **Scope:** most affected module or directory (optional but preferred)
- **Subject:** imperative mood, max 72 chars, no period at end
- **Body:** 1–2 sentences explaining **why**, not what. Omit for trivial changes.
- **Footer:** `BREAKING CHANGE: <description>` if applicable

Match the existing commit style from `git log`.

### 4. Confirm with User

Present:
- List of staged files
- Proposed commit message

Ask the user to **Commit**, **Edit message**, or **Cancel**.

### 5. Create Commit

Always use HEREDOC to preserve multiline formatting:

```bash
git commit -m "$(cat <<'EOF'
<type>(<scope>): <subject>

<body>
EOF
)"
```

### 6. Handle Pre-commit Hook Failure

If the commit fails due to a pre-commit hook:

1. Read the error output
2. Fix the issues (lint, format, type-check — whatever failed)
3. Stage the fixes: `git add <fixed-files>`
4. Create a **new** commit — never `--amend` (the failed commit didn't land, amending would modify the previous one)
5. Never use `--no-verify`

### 7. Verify

```bash
git status
git log -1
```

Show the created commit to the user.

## Rules

- **No AI attribution.** Commit messages must not mention Claude, AI, LLM, "generated by", "Co-Authored-By: Claude", or anything similar. Commits appear as regular developer work.
- Never use `git commit --no-verify` to bypass hooks.
- Never use `git commit --amend` after a hook failure.
- Focus the message on **why**, not **what** — the diff already shows the what.
- Keep the first line under 72 characters. Use the body for details.

## Examples

```
fix(store): prevent stale content after version switch

Remove optimistic update from setActiveVersion mutation — version content
is unavailable client-side at mutation time, causing desync between
TanStack Query cache and Zustand store.
```

```
feat(users): add pagination to users table
```

```
refactor(reports): extract report section store factory
```

```
chore(deps): upgrade TanStack Query to v5

BREAKING CHANGE: queryClient.invalidateQueries API changed — all
invalidation calls updated to new object syntax.
```
