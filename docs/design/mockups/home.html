the workflow

A workflow out of the box.

Marvin structures the whole lifecycle — from a requirement to a merged pull request. The critical steps are deterministic tools, not free-form prompting: the spec gate refuses an incomplete contract, the verify gate refuses a red build.

spec→ implement→ verify→ deliver→ learn ↺

The loop closes: lessons captured at delivery are recalled at the next spec.

commit & pr-* sec-* adr-* refactor-* track-* Tour the pipeline →
day one · pull requests

PRs that read well

Structured description, verification checklist, task links — your project history becomes easy to read from the first commit.

day one · requirements

Requirements become specs

Marvin formalizes a requirement into a spec in your repo. Leave it, come back, run it — the spec outlives the session.

day one · orientation

Never lose the plot

One dashboard over the board, artifacts, lessons, and decisions — you always know what is happening in the project.

artifacts

Your project keeps the receipts.

Everything marvin produces lives in one hidden directory, versioned with your code. No SaaS, no accounts, nothing to sync — clone the repo, get the whole history of specs, reports, and decisions.

.marvin/ — one directory, the whole story

.marvin/ ├─ task/ specs + verification ├─ track/ board tasks ├─ security/ scan reports ├─ refactor/ findings + plans ├─ memory/ team lessons ├─ handoff/ session handoffs └─ config.json
memory

Memory your team inherits

Lessons are captured when work ships and when bugs are debugged — then recalled the next time a task starts.

captured at deliver / debug→ .marvin/memory/→ recalled at task-start

Because memory is files in the repo, it travels through git: one developer's lesson becomes the whole team's default.

No server. No account. Just files your team already shares.
