marvin's committed `plugins/marvin/widgets/*.html` are real MCP Apps views and can be embedded live in any page WITHOUT importing `@modelcontextprotocol/ext-apps` — the site does it in ~150 lines (`packages/site/src/lib/widget-host.ts`). Why it works: production widgets mount with no seam, so they run `useApp()`, which hard-wires a `PostMessageTransport` at `window.parent`; that transport sends with `postMessage(msg, "*")` and validates the peer by `event.source` (window identity), NOT by origin, so a sandboxed frame with an opaque origin still connects, and `event.data` IS the raw JSON-RPC message (no envelope). Reason to hand-roll rather than import AppBridge: the SDK bundle is ~95% zod and is exactly what makes each widget document ~300 KB — importing it into a host page undoes any low-JS budget. The required surface is tiny: answer `ui/initialize` (result needs ALL FOUR of protocolVersion / hostInfo / hostCapabilities / hostContext; there is no version negotiation, so echo the requested version), then on `ui/notifications/initialized` send `ui/notifications/tool-input` followed by `ui/notifications/tool-result`. THREE TRAPS, each of which fails silently: (1) `tool-result` params `required: ["content"]` — omit `content` alongside `structuredContent` and the view's zod parse fails, the SDK SWALLOWS notification-handler errors, and the frame sits on "Connecting…" with NOTHING in the console; (2) the widgets issue two host-bound REQUESTS — `ui/open-link` (lib/links.ts) and `ui/message` (HandoffsWidget, ReportsWidget Sync) — which hang on the SDK's 60s request timeout unless answered, so answer both with `{}` even if the controls are meant to be inert; (3) the widgets IGNORE the protocol's theme channel entirely (no `onhostcontextchanged`/`applyDocumentTheme` anywhere in packages/marvin-widgets/src), so `ui/notifications/host-context-changed` is inert — to force a theme the parent must set `data-theme` on the framed `.mvroot` element (the token override selector is `.mvroot[data-theme="dark"]`, tokens.ts:76-77) AND `color-scheme` on its documentElement (the `:root{color-scheme:light dark}` in each widget head is outside the `.mvroot` scope, so scrollbars/controls otherwise stay on the OS scheme). That reach-in requires `sandbox="allow-scripts allow-same-origin"`; the strict-sandbox alternative is teaching the family host-context theming, which rebuilds 9 committed HTML past verify-widgets and invalidates 101 darwin visual baselines. Also: a `display:none` container still boots the frame fine, but `ResizeObserver` reports 0×0 so `size-changed` arrives with height 0 — guard `height > 0` or the frame is stuck at its CSS min-height forever. Fixtures need no new plumbing: 8 of 9 `packages/marvin-widgets/src/widgets/*/fixture.ts` are type-only imports, so gen-catalog's in-memory `ts.transpileModule` → data:-URL trick reads them unmodified; only `help/fixture.ts` has a runtime import needing specifier rewrite. Because the protocol is hand-rolled, the e2e MUST assert on fixture-only data through `frameLocator` (e.g. the help fixture's project name `telegram-publications`, which appears 0 times in help.html) — that is what makes the suite a drift detector instead of an "iframe mounted" check.
